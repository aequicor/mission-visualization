# План: читаемость UML/ER-диаграмм (публикационное качество)

Статус: **P0 реализована (2026-07-16), P1–P5 впереди.** Приоритет типов диаграмм по итогам
обсуждения — **все** (ER, class/UML, flowchart/activity/state, use-case; sequence не трогаем).

Что вошло в P0 (сверх плана — по итогам adversarial-ревью):
- ER-роутинг унифицирован с ORTHOGONAL полностью: grid A* + facing-side планирование +
  разнос якорей; **стаб ER = obstacleMargin** (опция `entityRelationStub` удалена — стаб
  длиннее коридора давал «усы»-развороты у якорей, т.к. цель A* не ограничивает направление
  прихода).
- Line-jumps: дефолт `jumps arc`, `corners rounded` (модель/CNL/JSON/доки/скиллы синхронно);
  прыгает только верхнее ребро (по layer-порядку отрисовки), канва и SVG-экспорт согласованы.
- Подпись MIDDLE соскальзывает с пересечений (кандидаты 0.5→0.42→0.58→…); обстакли — только
  видимые слои; drag-база подписи — **несдвинутый** якорь (иначе подпись прыгает при драге);
  SVG-экспорт использует общий `edgeLabelAnchorPoint`.
- LayeredLayout: связные компоненты раскладываются отдельно и shelf-пакуются; barycenter
  сидится BFS-обходом и крутится до сходимости (макс. 8 раундов).
- Авто-layout при вставке шаблонов уже был (шаблоны запечены через `autoLayout`) — кода
  не потребовалось.

Цель: схемы читаются как в качественных научных работах — минимум/ноль пересечений рёбер,
прямые выровненные связи, ровные равномерные отступы, рёбра не режут узлы, аккуратные подписи.

Модуль: `:subsystems:diagrams` (чистый Kotlin, KMP, **детерминированный**), рендерер —
`:subsystems:diagrams-compose`. Авторинг/write-back — CNL (`*.layout.md`), позиции узлов —
явные `position x y`.

---

## TL;DR

- **Роутер уже сильный** (A\* по сетке, обход препятствий, разнос якорей, разведение полос,
  line-jumps, скругления, сплайны). Узкое место — **раскладка узлов**, а не роутинг рёбер.
- **Для конкретной ручной ER-схемы** (как на скриншоте) самый большой рычаг — **не** раскладка, а
  **дыра в ER-роутинге**: `ENTITY_RELATION`-рёбра идут без обхода препятствий и без разноса якорей.
  Это чинится на уровне роутера, без всякой авто-раскладки → фаза **P0**.
- **Для качества в целом** самый большой рычаг — **dummy-узлы** в настоящем Sugiyama (P1): без них
  ни минимизация пересечений (P2), ни выпрямление координат (P3) работать не могут.
- Архитектура — **гибрид**: Sugiyama по умолчанию для иерархических типов + SMACOF force-layout для
  циклических ER + общий VPSC-солвер; всё кормит **существующий** роутер, запись в SLM остаётся
  positions-only (CNL-контракт не меняется).

## Диагноз (по коду)

### Уровень 1 — раскладка узлов (главная причина «непубликационности»)

Имеющийся `layeredLayout` — «Sugiyama-lite» и структурно **не умеет** минимизировать пересечения:

- **Нет dummy/virtual-узлов** для рёбер, пересекающих ≥2 слоя
  (`LayeredLayout.kt:37-45`, `:55-74`). Длинные рёбра невидимы для barycenter-упорядочивания и
  не резервируют коридор → либо сильно петляют (лишние пересечения), либо режут промежуточные
  узлы. Все четыре агента ставят это причиной №1.
- **Нет метрики пересечений / transpose / keep-best** (`LayeredLayout.kt:71-74`): всего 2 barycenter-
  прохода без подсчёта, пересечения по-настоящему не минимизируются, плохой проход не откатывается.
- **Нет выпрямления координат** (`LayeredLayout.kt:82-105`): каждый слой центрируется независимо на
  равных промежутках → межслойные рёбра почти никогда не коллинеарны → «волнистые» зигзаги даже
  когда пересечений нет. Это эстетика, не счётчик пересечений.
- **Раскладка слепа к типу диаграммы и направлению** (`AutoLayout.kt:19` — только forest-vs-DAG;
  `LayoutConfig.kt:32` — дефолт TOP_DOWN): ER/циклические графы загоняются в ложную longest-path
  иерархию; DFS cycle-break (`LayoutScopes.kt:66-87`) выдумывает произвольный ярус, который роутер
  спасти не может.

### Уровень 2 — роутинг именно ER-рёбер (чинит ручную схему без раскладки)

- `routeEntityRelation` (`EdgeRouter.kt:738-769`) **не обходит препятствия** — использует
  `manhattanLeg`, а не A\*-сетку `OrthogonalGrid`, как `ORTHOGONAL`. → ER-связи режут промежуточные
  сущности.
- `resolveEntityRelationEnd` FloatingAnchor (`EdgeRouter.kt:784-789`) прибивает якорь к `centerY`,
  только LEFT/RIGHT → несколько связей к одной сущности сходятся в одну точку.
- `sideAttachments` (`EdgeRouter.kt:469-470`) фильтрует только ORTHOGONAL/SIMPLE → ER **исключён**
  из разноса якорей.

Итог: «стрелки друг на друге» на скриншоте — во многом ER-роутинг, независимо от расстановки.

### Уровень 3 — дефолты полировки выключены

- `LineJumpStyle = NONE` и `cornerStyle = SHARP` везде (`RoutingOptions`/`DiagramEdge:198`/
  `DiagramStyle:49`). Где линии всё же пересекаются — голое наложение с жёсткими углами вместо
  арок-перескоков и скруглений (всё уже реализовано в `RoutedEdgePath.kt`).
- Подписи рёбер стоят на фиксированной доле длины дуги без обхода коллизий
  (`DiagramEdgeRenderer.kt:110-123`).
- Авто-layout — opt-in/one-shot (`DiagramEditing.kt:305-306`, интент `ApplyDiagramAutoLayout`):
  ручная схема никогда не переставляется, любой апгрейд раскладки для неё инертен, пока не нажать кнопку.
- Несвязные компоненты сваливаются в слой 0 без упаковки (`LayeredLayout.kt:38-41`).

## Целевая архитектура — гибрид

Один движок на все 7 типов не выигрывает. Решение:

- **Sugiyama** (dummy-узлы → median+transpose+keep-best → Brandes–Köpf) — **по умолчанию** для
  иерархических: class/UML, component/deployment, state, activity, use-case, flowchart.
- **SMACOF stress-majorization** (classical-MDS init + VPSC overlap removal + axis-snap) — **fallback**
  для плотных/циклических ER и ненаправленных many-to-many, где Sugiyama выдумывает ложную иерархию.
- **VPSC** (1-D separation-constraint solver) — общий примитив: и для non-overlap в Brandes–Köpf, и
  для opt-in кнопки **«Tidy/Align»** (снап строк/колонок ручных диаграмм без полной перекладки).
- **Sequence** остаётся на своей `SequenceRouting`.
- **Роутер переиспользуется как есть**; ER дополнительно получает обход препятствий + разнос якорей
  на уровне роутинга (независимо от раскладки).

AUTO-классификатор `classifyTopology(graph, scope) → TREE | LAYERED | FORCE`
(ER/ненаправленный/высокая цикломатичность → FORCE).

**Инварианты, которые нельзя ломать:**
- Всё идёт через существующий шов `positionsFor → layoutHierarchically` (`LayoutScopes.kt:104-146`):
  контейнерная рекурсия, сдвиг поддеревьев и positions-only write-back в SLM не трогаются.
- **Детерминизм**: id-сортировка итераций, производные id для dummy, keep-best с лексикографическим
  tie-break, MDS power-iteration с index-seeded стартом (без `Math.random`/`Date.now`), **integer
  grid-snap** как стабилизатор FP-дрейфа JVM↔wasm, чтобы записанные в CNL координаты были bit-stable.

### P0 — Быстрые победы (роутер + workflow, без новой раскладки) · M

Чинит именно ручную ER-жалобу и даёт дешёвую глобальную опрятность до перестройки раскладки.

- Роутить `ENTITY_RELATION` через `OrthogonalGrid` A\* как `ORTHOGONAL` (добавить `foreignBounds`/сетку
  в `routeEntityRelation`).
- Включить ER в `sideAttachments`-разнос и разрешить TOP/BOTTOM (не только `centerY` LEFT/RIGHT).
- Line-jumps (ARC/GAP) и скругления по умолчанию на пересечениях.
- Авто-layout-при-вставке шаблонов (паритет с импортом текста) — новые диаграммы стартуют опрятными.
- Packing связных компонент, чтобы несвязные/source-узлы не сваливались в слой 0 и не растягивали схему.
- Небольшой сдвиг плашки подписи с плотных коридоров/пересечений.
- Дешёвый бонус: barycenter до сходимости (или больше проходов) + сид DFS/BFS вместо id-порядка.

Файлы: `routing/EdgeRouter.kt` (`routeEntityRelation:738-769`, `resolveEntityRelationEnd:784-789`,
`sideAttachments:469-470`); `layout/LayeredLayout.kt` (split компонент в слое 0);
`diagrams-compose/DiagramEdgeRenderer.kt` (`:110-123` сдвиг подписи, дефолт line-jumps);
`shared/.../editor/presentation/DiagramEditing.kt` (`:305` авто-layout шаблонов).

Проверка: юнит — ER-фикстура «сущность между двумя связанными» утверждает, что путь не пересекает
bounds средней сущности; «много связей у одной сущности» — различные координаты якорей. wasm-визуалка
на проблемной ER-схеме.

### P1 — Ядро Sugiyama: dummy-узлы + VPSC · M

Длинные рёбра перестают резать узлы; строится переиспользуемый separation-солвер.

- Внутренний `LayeredGraph` (реальные + dummy `LNode`/`LEdge`, порты).
- Вынести longest-path layering; вставка dummy-цепочек на каждый промежуточный ранг с фиксированным
  cross-extent (резервирует коридор).
- Детерминированный VPSC 1-D active-set separation-солвер.
- Опционально: «via»-waypoints по dummy-цепочке за флагом (пинятся существующим `waypointedEdgeIds`).

Файлы (новые): `layout/LayeredGraph.kt`, `layout/DummyNodes.kt`, `layout/LayerAssignment.kt`,
`layout/Vpsc.kt`. Правки: `layout/LayeredLayout.kt` (`layeredPositions` → оркестратор),
`layout/LayoutConfig.kt` (`dummySize`, `emitLongEdgeWaypoints`), `layout/LayoutScopes.kt`.

Проверка: юнит — для ребра 0→3 ни один реальный узел не лежит в коридоре ребра и ребро учтено в
упорядочивании; `VpscTest` — нулевое перекрытие + `nodeGap` + run-twice-equal.

### P2 — Минимизация пересечений · M

Пересечения реально минимизируются, а не просто двигаются узлы.

- Weighted-median (down/up) + transpose (обмен соседей, пока точный счётчик строго падает) +
  keep-best по раундам. Счёт пересечений — аккумулятор Barth–Mutzel–Jünger O(E log V). Tie-break
  лексикографически по id. Заменяет 2-проходный barycenter-блок.

Файлы (новые): `layout/CrossingMinimization.kt`, `layout/CrossCount.kt` + тест. Правка:
`layout/LayeredLayout.kt`.

Проверка: golden-тесты — счётчик пересечений на diamond/dense-class/flowchart-фикстурах ≤ baseline
barycenter и стабилен между двумя прогонами.

### P3 — Brandes–Köpf выпрямление координат · L

Прямые, выровненные, равномерные рёбра — публикационный вид.

- Brandes–Köpf 4-alignment (up/down × left/right) + block-class компакция + median-balance;
  внутрислойный non-overlap через VPSC (P1); dummy-цепочки выпрямляются в вертикальные прогоны
  (почти нулевой изгиб длинных рёбер); одиночный родитель выравнивается над ребёнком.
- Priority-method как fallback за config-enum.

Файлы: правка `layout/LayeredLayout.kt` (переразводка L1→L6, сохраняя per-scope иерархию).

Проверка: юнит — межслойные рёбра выровненного родителя/ребёнка коллинеарны (равная cross-координата);
dummy-цепочка длинного ребра делит один x. wasm-визуалка на class/state/flowchart-шаблонах.

### P4 — ER stress-fallback + AUTO-классификатор + Tidy/Align · L

Циклическому ER — органичная раскладка вместо ложной иерархии; помощь ручным диаграммам без полной перекладки.

- SMACOF (classical-MDS power-iteration init, index-seeded, фикс. лимит итераций) + VPSC overlap
  removal (P1) + 1-D axis-union snap + integer grid-snap.
- `classifyTopology(graph, scope) → TREE|LAYERED|FORCE`.
- Opt-in интент «Tidy/Align»: снап строк/колонок ручных узлов через VPSC, topology-preserving.

Проверка: run-twice-equal на снапнутых int; stress монотонно падает; ноль перекрытий после removal;
AUTO выбирает FORCE для ER-шаблона и циклической сетки, LAYERED для diamond-DAG, TREE для fan-леса.
wasm-визуалка на ER-шаблоне.

### P5 — Пресеты, ориентация, ER-полировка · S

- Пресет «Publication» через `DiagramLayoutIntent`; ориентация по типу связи
  (Generalization/Realization: родитель получает меньший ранг → наследование указывает вверх);
  LEFT_RIGHT по умолчанию для ER auto.
- Проверить кооперацию dummy-waypoints с `EdgeNudging` + line-jumps.
- Документация в `engine/README.md` + `EDITOR.md`.

Проверка: юнит — фикстура наследования утверждает `rank(родитель) < rank(ребёнок)`; combined-тест —
пинованное длинное ребро в общем коридоре с движимым не перекрывается. wasm-визуалка по всем типам.

## Рекомендации по типам диаграмм

- **ER / сущность-связь.** Главная боль — **роутер**, не раскладка. Сначала P0 (obstacle-avoidance +
  anchor-spreading) — чинит ручную схему. Для авто-расстановки — SMACOF+VPSC FORCE (P4), **не**
  Sugiyama (у циклических схем нет честной иерархии). По умолчанию LEFT_RIGHT. Предложить «Tidy/Align».
- **class / UML.** Полный Sugiyama (dummy + crossing-min + Brandes–Köpf), TOP_DOWN. Ориентировать
  Generalization/Realization так, чтобы родитель получал меньший ранг (наследование вверх). Идеальная
  зона слоёного движка.
- **state / activity.** Sugiyama с DFS cycle-break для петель состояний; TOP_DOWN для state,
  LEFT_RIGHT для activity/swimlanes. Нужна обработка self-loop (боковые порты), чтобы петли не портили
  упорядочивание.
- **use-case.** Небольшие почти-двудольные графы: слоёный LEFT_RIGHT, актёры прижаты к левому/правому
  краю, use-cases в центре; SMACOF-fallback приемлем для запутанных. Низкий приоритет — малые графы уже
  роутятся сносно.
- **sequence.** Оставить `SequenceRouting` как есть — **не** гнать через Sugiyama/SMACOF. Улучшения
  только: равные промежутки lifeline и порядок lifeline по первому появлению сообщения.
- **flowchart.** Sugiyama DAG, TOP_DOWN — чистейший фит. Подписи ветвлений напрямую выигрывают от P0
  сдвига коллизий.

## Открытые решения (перед реализацией)

1. **Waypoints длинных рёбер в CNL.** Персистить как `via (x y)` (честно, draw.io-подобно, но раздувает
   исходник и связывает авто-раскладку с write-back) **или** держать только in-memory (чистый исходник,
   но ручная правка позже перероутит без коридора и может визуально разойтись)? — решение по scope P1/P4.
2. **AUTO авто-классифицирует ER/цикл → FORCE** или движок остаётся явным выбором пользователя? Ошибка
   классификации плотного DAG в force (или наоборот) деградирует качество; пороги надо калибровать на
   реальных ER/flowchart/class-шаблонах.
3. **Авто-применять раскладку на живой поверхности** или строго opt-in за кнопкой? ER-жалоба чинится
   расстановкой только при явном вызове, но авто-применение конфликтует с ручной расстановкой и
   minimal-diff write-back. (P0-фиксы роутера обходят это для ER.)
4. **Детерминизм stress-движка.** Достаточно integer grid-snap против FP-дрейфа JVM↔wasm (sin/cos,
   неассоциативная сумма), или держать ER на чисто целочисленной эвристике, избегая FP вовсе? — решает
   жизнеспособность SMACOF-ветки.
5. **Объём конфигурации.** Выставлять `layeringMode`/`coordinateAssignment`/`crossingRounds`/
   `direction-per-kind` или только пара заблокированных пресетов («Publication», «Compact»), чтобы не
   плодить непротестированные комбинации?
6. **«Tidy/Align» отдельной второй кнопкой** (помогает всем ручным диаграммам чистой геометрией) или
   одна кнопка «Авто-раскладка» — вся продуктовая поверхность?

## Приложение: почему гибрид, а не один движок

- **Sugiyama** — публикационное качество для DAG-образных UML, но явно плох для циклического ER
  (ровно тот граф, на который жалоба).
- **SMACOF** (наивысший fit по панели) — верная цель для ER/циклов (его cost-функция и есть равномерная
  структурно-верная раскладка, монотонна → терминируется под фикс. лимитом без random-сида), но слаб на
  честно иерархических данных (нет ярусов, нет минимизации пересечений).
- **Чисто-ортогональный constraint-роутер** (HOLA/libavoid) — XL-объём (полный channel-router +
  перепроводка nudging), переинвестиция; но две его идеи несут нагрузку и вошли в гибрид: переиспользуемый
  **VPSC** и opt-in **«Tidy/Align»** для ручных диаграмм.

Гибрид берёт Sugiyama-для-иерархий + SMACOF-для-ER + VPSC-как-общий-примитив и признаёт, что ER-боль
частично — **роутерная** и чинится без изменения расстановки (P0).
